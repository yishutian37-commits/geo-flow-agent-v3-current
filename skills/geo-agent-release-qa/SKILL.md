# GEO Agent Release QA

用于处理用户要求“全面审查项目”“修复逻辑断链”“按优先级补齐”“重新打包”“清理无用文件”“发版前检查”等任务。

## 工作原则

- 先保护现场，再修改。重大改动前确认或创建备份。
- 先读代码和现有测试，再下结论。
- 修复必须覆盖数据链路、异常分支、接口连接、状态刷新。
- 每轮完成后必须跑测试和构建，打包请求必须检查 release 产物。
- 不回滚用户已有改动，不清理无法确认用途的文件。

## 固定流程

1. 定位项目根目录，确认关键目录：
   - `backend/app`
   - `backend/tests`
   - `frontend/src`
   - `desktop`
   - `_external/QWebBridge`
   - `release`
2. 快速体检：
   - `git status --short`
   - 搜索最近相关模块和接口。
   - 检查后端路由是否重复。
3. 逻辑审查顺序：
   - 项目管理链路：项目创建、编辑、删除、关联数据清理。
   - 品牌事实链路：资料上传、AI 抽取、确认、引用。
   - 问题库链路：生成、启停、删除、检测引用。
   - 监测链路：平台、问题、样本、原始回答、截图、来源、报告。
   - 内容链路：任务、草稿、反馈、记忆、发布检查、发布记录。
   - 桌面链路：后端启动、前端入口、QWebBridge、打包资源。
4. 修复要求：
   - 后端修复要补测试。
   - 前端修复要确认 API 调用、状态刷新、错误提示。
   - 删除逻辑必须清理关联表和本地证据文件。
5. 验证命令：
   - 后端：`.\.venv\Scripts\python.exe -m pytest tests -q`
   - 前端：`npm.cmd --prefix frontend run build`
   - 打包：`npm.cmd run dist:win`
   - 产物：`Get-ChildItem release`

## 质量门槛

- 后端测试通过。
- 前端构建通过。
- 若用户要求桌面包，必须生成或更新 `release` 下安装包和便携包。
- 最终报告必须列出：
  - 发现的问题
  - 修复文件
  - 测试结果
  - 打包产物路径
  - 剩余风险

## 输出格式

优先用简洁中文汇报：

- 已修复什么。
- 如何验证。
- 新包在哪里。
- 哪些问题只是维护项，不影响当前使用。
