# GEO Agent Release QA

用于处理用户要求“全面审查项目”“修复逻辑断链”“按优先级补齐”“根据 PRD 看还有什么没实现”“重新打包”“清理无用文件”“发版前检查”“更新 GitHub 仓库”等任务。

## 产品判断核心

这个项目不是普通内容生成器，而是 GEO 证据链与复测决策系统。审查或新增功能时，先判断它是否增强这条主线：

`品牌事实 -> 问题库 -> 多平台检测 -> 原始回答/截图/来源留档 -> 提及/推荐/情绪判断 -> 内容优化任务 -> 平台发文 -> 发布记录 -> 复测报告`

如果新功能不能增强“证据可信、判断可解释、发布可复测、经验可沉淀”，优先级通常不应高于主链路缺口。

## 工作原则

- 先保护现场，再修改。重大改动前确认或创建备份。
- 先读代码和现有测试，再下结论。
- 修复必须覆盖数据链路、异常分支、接口连接、状态刷新。
- 每轮完成后必须跑测试和构建，打包请求必须检查 release 产物。
- 不回滚用户已有改动，不清理无法确认用途的文件。
- 更新 GitHub 时不要强推 `main`；优先推分支、建 PR、解决冲突后合并。

## 固定流程

1. 定位项目根目录，确认关键目录：
   - `backend/app`
   - `backend/tests`
   - `frontend/src`
   - `desktop`
   - `_external/QWebBridge`
   - `release`
2. 快速体检：
   - `git status --short`
   - 搜索最近相关模块和接口。
   - 检查后端路由是否重复。
3. 逻辑审查顺序：
   - 项目管理链路：项目创建、编辑、删除、关联数据清理。
   - 品牌事实链路：资料上传、AI 抽取、确认、引用。
   - 问题库链路：生成、启停、删除、检测引用。
   - 监测链路：平台、问题、样本、原始回答、截图、来源、报告。
   - 内容链路：任务、草稿、反馈、记忆、发布检查、发布记录。
   - 桌面链路：后端启动、前端入口、QWebBridge、打包资源。
4. 断链判断：
   - 页面能操作但数据没有写入后端。
   - 后端保存了数据但前端没有刷新。
   - 上游生成了内容但下游无法选择或引用。
   - 结论存在但没有原始证据。
   - 删除了主记录但子表、截图、来源、报告仍残留。
   - AI 反馈被原话保存，没有总结成可执行规则。
   - 桌面包没有包含最新前后端构建结果。
5. 修复要求：
   - 后端修复要补测试。
   - 前端修复要确认 API 调用、状态刷新、错误提示。
   - 删除逻辑必须清理关联表和本地证据文件。
6. 验证命令：
   - 后端：`.\.venv\Scripts\python.exe -m pytest tests -q`
   - 前端：`npm.cmd --prefix frontend run build`
   - 打包：`npm.cmd run dist:win`
   - 产物：`Get-ChildItem release`
7. GitHub 更新：
   - 先更新 README、应用介绍、使用说明等文档。
   - 先 `git fetch origin main`，确认远程是否有新提交。
   - 推送到工作分支，例如 `codex/<topic>`。
   - 创建 PR 合并，不直接强推 `main`。
   - 如 GitHub 网页显示冲突，优先本地解决；网络推送不可用且冲突很少时，可用 WebBridge 的冲突编辑器兜底。

## 优先级规则

P0：
- 影响项目、事实、问题、检测、内容、报告、发布、复测主链路。
- 检测结论没有原始回答、截图、来源或判断依据。
- 数据删除、更新、重生成会累加或残留错误数据。
- 桌面应用无法启动、打包产物不是最新代码。

P1：
- 业务人员看不懂页面或报告。
- 多平台、多批次、平台版本、发布记录聚合不完整。
- 记忆库、行业模板、问题矩阵沉淀不顺。

P2：
- 自动化体验增强、图表美化、定时任务、竞品扩展、发布辅助。

## 质量门槛

- 后端测试通过。
- 前端构建通过。
- 若用户要求桌面包，必须生成或更新 `release` 下安装包和便携包。
- 若用户要求上传 GitHub，必须说明 PR 或合并链接，以及本地是否已对齐远程。
- 最终报告必须列出：
  - 发现的问题
  - 修复文件
  - 测试结果
  - 打包产物路径
  - 剩余风险

## 输出格式

优先用简洁中文汇报：

- 已修复什么。
- 如何验证。
- 新包在哪里。
- 哪些问题只是维护项，不影响当前使用。
