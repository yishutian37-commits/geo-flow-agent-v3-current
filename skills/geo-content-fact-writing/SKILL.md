# GEO Content Fact Writing

用于处理内容管理、文章生成、草稿改写、事实引用、用户反馈记忆、标题正文拆分、合规检查等任务。

## 核心原则

内容生成必须由已确认品牌事实驱动，不能让文章和企业信息断开。

标准链路：

`品牌事实库 -> 问题意图 -> 内容任务 -> 平台类型 -> 标题/正文结构化生成 -> 事实校验 -> 合规检查 -> 用户反馈 -> AI 总结规则 -> 记忆库 -> 二次改写`

## 输入要求

生成文章前必须收集：

- 项目和品牌主体
- 已确认品牌事实
- 关联问题
- 内容类型
- 目标平台
- 写作目标
- 用户反馈记忆

如果没有可公开使用的已确认事实，应提示用户先补充事实，不生成模板占位文章。

## 输出结构要求

模型可以内部返回 JSON，但最终入库必须拆成明确字段：

- `title`
- `body`
- `summary`
- `platform_notes`
- `fact_refs`
- `compliance_check`

用户可见正文中禁止出现：

- JSON 原文
- `[品牌名]` 等占位符
- `[行业]` 等模板字段
- `[FACT_REFS]`
- `[COMPLIANCE_CHECK]`
- 模型解释性废话

事实引用可以内部保存，但不要直接混进正文。

## 标题处理

如果模型返回 `title_candidates`：

1. 选择最适合内容任务的一条作为标题。
2. 写入标题字段。
3. 不要把候选标题数组塞进正文。

## 反馈记忆规则

用户原话不能直接变成规则。必须先由 AI 总结成清晰、可执行的写作规则。

例：

用户反馈：`字数不要低于1500字，少一些ai味道`

应沉淀为：

- 长期规则：正文不少于 1500 字。
- 语言风格：减少模板化表达，增加自然叙述和真实场景。

## 合规检查

合规检查应该是提示项，不应无条件阻止用户记录已发布结果。

如果草稿高风险：

- 生成明确风险说明。
- 允许用户选择“仍然记录发布结果”。
- 保存风险状态，便于后续复盘。

## 验收清单

- 标题框有标题。
- 正文是纯正文，不是 JSON。
- 没有模板占位符。
- 事实均来自已确认事实库。
- 反馈记忆会参与下一次生成。
- 发布记录支持多平台或手动平台名称。
- 生成失败时显示可理解错误，不显示 `[object Object]`。
