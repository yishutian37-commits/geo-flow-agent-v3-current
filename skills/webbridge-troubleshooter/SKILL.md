# WebBridge Troubleshooter

用于处理 Kimi WebBridge / QWebBridge 相关故障，包括插件未就绪、502、无法发送、无法抓回答、截图不完整、视觉模式失败等。

## 先判断是哪一层坏了

按顺序排查：

1. 桥服务是否启动。
2. 浏览器插件是否安装并连接。
3. 目标 AI 网页是否已登录。
4. 输入框 selector 是否可用。
5. 发送按钮 selector 是否可用。
6. 回答是否已生成完成。
7. 回答容器是否能被 DOM 抓取。
8. 截图是否截到有效证据。

## 常见端口

- Kimi WebBridge：`http://127.0.0.1:10086`
- QWebBridge：`http://127.0.0.1:10087`

健康检查失败时，不要直接假设代码错。先确认插件和服务连接。

## 常见 502 原因

- 浏览器扩展未安装或未连接。
- 桥服务启动了，但扩展没有响应。
- Kimi WebBridge 守护进程停止，或 `daemon.pid` 是旧进程。
- Kimi WebBridge 启动时日志轮转失败，例如 `daemon.log` 被锁导致 `Access is denied`。
- 当前标签页被关闭。
- 目标网页未登录或在验证。
- 页面结构变化，selector 不匹配。
- 发送按钮存在但禁用。
- 回答还在生成，提前切到下一题。

## 发送失败排查

当页面已输入问题但未发送：

1. 先尝试 Enter / Ctrl+Enter。
2. 再使用平台配置的 `submit_selector`。
3. 再扫描候选按钮。
4. 候选按钮应返回 selector、label、score。
5. 如果输入框仍保留原问题，应判定发送失败并停止本次检测，不继续下一题。

## Kimi WebBridge 守护进程排查

当 `http://127.0.0.1:10086/command` 无法连接或 `status` 显示 `running:false`：

1. 先运行：
   - `%USERPROFILE%\.kimi-webbridge\bin\kimi-webbridge.exe status`
2. 如果启动报 `rotate log ... Access is denied`：
   - 检查是否有旧进程占用日志。
   - 清理 stale `daemon.pid`。
   - 将 `logs/daemon.log` 移到带时间戳的备份名。
   - 再重新 `start`。
3. 如果 `running:true` 但 `extension_connected:false`：
   - 等待几秒重查。
   - 确认浏览器扩展已启用。
   - 打开一个真实网页触发扩展连接。
4. 版本提示可记录，但不要在任务中途随意升级，避免引入新变量。

## 等待回答策略

不要只等待固定秒数。优先使用组合策略：

- 等待回答容器出现。
- 等待回答文本长度连续几次不再变化。
- 等待停止按钮消失或发送按钮恢复。
- 用户配置的等待时间作为上限，不是唯一依据。

## 文本模式和视觉模式

默认使用文本抓取模式。

视觉识别模式只有用户明确开启时使用。开启视觉模式后：

- 文本 DOM 抓取不作为主判断。
- 截图送入多模态模型分析。
- 模型必须返回可解析结构。
- 如果视觉模型无图片理解能力，应明确提示并回退或要求切换模型。

## 截图策略

优先级：

1. 回答容器完整截图。
2. 品牌提及位置截图。
3. 当前视口截图。

如果回答很长，至少要保存：

- 完整文本。
- 品牌提及附近文本。
- 品牌提及区域截图。

不要只截最后一个视口。若品牌在回答开头被提及，应定位到首次提及位置后截图上下文。

## 修复后验证

- 能打开目标 AI 平台。
- 能输入问题。
- 能成功点击发送。
- 回答生成完成后再切下一题。
- 能保存回答文本。
- 能保存截图。
- 失败时错误信息能说明是哪一层失败。
