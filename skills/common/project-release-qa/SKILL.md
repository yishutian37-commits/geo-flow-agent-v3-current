# 通用项目审查与发版质检

当用户要求对任意软件项目进行“全面审查”“修复逻辑断链”“清理无用文件”“测试验证”“构建打包”“发版前检查”时，使用这个 skill。

## 核心原则

不要预设项目技术栈。先读取仓库结构，再从项目文件中判断测试命令、构建命令、打包命令和质量门槛。

## 结构识别清单

优先识别这些文件和目录：

- 技术栈文件：`package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`、`pom.xml`、`build.gradle`、`Makefile`、Docker 配置。
- 源码目录。
- 测试目录。
- 构建产物目录。
- 打包脚本。
- CI 或发版脚本。
- 本地环境文件，避免误改。

## 标准流程

1. 保护当前状态：
   - 如果是 Git 仓库，先查看 `git status --short`。
   - 不回滚用户已有改动。
   - 涉及大范围或高风险改动时，先备份或确认备份策略。
2. 审查架构：
   - 启动入口。
   - API 路由。
   - 数据模型。
   - 状态管理。
   - 前后端边界。
   - 桌面端、移动端或打包壳。
3. 识别逻辑断点：
   - 模块间数据没有传递下去。
   - 条件分支缺少边界处理。
   - API 契约前后端不一致。
   - 变更后页面状态没有刷新。
   - 删除或更新操作留下孤儿数据。
   - 异常处理过于粗糙。
4. 小范围修复：
   - 遵循项目已有风格。
   - 修改业务逻辑时补测试。
   - 不做无关重构。
5. 验证：
   - 后端逻辑改动跑后端测试。
   - 前端改动跑前端构建或相关测试。
   - 涉及发版时跑打包命令。
   - 可行时做关键流程冒烟测试。
6. 汇报：
   - 发现的问题。
   - 修改的文件。
   - 执行的测试和构建。
   - 发版产物路径。
   - 剩余风险。

## 命令识别规则

优先使用项目自己声明的命令：

- Node 项目：查看 `package.json` 中的 `test`、`build`、`dist`、`package`、`lint`。
- Python 项目：优先使用项目虚拟环境，再考虑 `pytest`、`tox`、`python -m unittest`。
- Rust 项目：`cargo test`、`cargo build`。
- Go 项目：`go test ./...`、`go build`。
- Java 项目：优先使用 Maven 或 Gradle wrapper。
- Docker 项目：只有确实需要时再用 compose。

如果命令耗时很长、可能破坏数据、需要联网或会启动 GUI，应按当前环境规则先说明并申请权限。

## 质量门槛

任务完成前，至少要做到：

- 修改过后端逻辑，就跑相关后端测试。
- 修改过前端界面，就跑前端构建或测试。
- 修改过打包或用户要求发版，就检查 release 产物。
- 如果测试无法运行，要说明原因和剩余风险。

## 最终汇报格式

优先用这几个小节：

- 已修复
- 验证结果
- 发版产物
- 剩余风险
