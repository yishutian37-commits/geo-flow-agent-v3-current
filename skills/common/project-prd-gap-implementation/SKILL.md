# 通用 PRD 对照开发

当用户要求“根据 PRD 看还有什么没实现”“按优先级补齐”“继续下一步”“对照需求文档施工”时，使用这个 skill。

## 核心原则

把 PRD 当作产品契约，但必须结合实际代码和当前行为验证。不要只凭文件名或主观猜测判断实现状态。

## 输入来源

需求可能来自：

- `.docx`、`.md`、`.pdf`、`.txt`
- issue 列表
- 对话中的需求描述
- 截图
- 当前应用行为

如果用户提到的文档在本地存在，应优先读取本地文件。

## 工作流程

1. 提取需求：
   - 功能模块。
   - 页面。
   - 用户角色。
   - 数据实体。
   - 业务流程。
   - 字段。
   - 权限。
   - 异常状态。
   - 报告或导出。
2. 建立实现映射：
   - 后端模型。
   - API 接口。
   - 服务层业务逻辑。
   - 前端页面和组件。
   - 状态管理。
   - 测试。
   - 打包或部署。
3. 标注状态：
   - 已实现。
   - 部分实现。
   - 未实现。
   - 已实现但偏离 PRD。
   - 已实现但实际不可用。
4. 排优先级：
   - P0：阻断主流程、数据完整性、安全性或发版。
   - P1：重要业务价值或高频工作流。
   - P2：增强体验、自动化、分析能力或细节优化。
5. 分批实现：
   - 每次只做一个清晰的功能包。
   - 补充或更新测试。
   - 每批结束后执行相关验证。
6. 汇报：
   - 本轮完成什么。
   - 还剩什么。
   - 下一步建议做什么。

## 缺口表格式

汇报时优先使用这个结构：

| 优先级 | PRD 要求 | 当前状态 | 缺口 | 建议修复 |
|---|---|---|---|---|

## 施工规则

- 除非用户明确要求，不要先做低优先级功能。
- 不擅自扩大或改变 PRD 范围。
- 保留用户已有数据和无关改动。
- 如果 PRD 与当前应用行为冲突，要指出冲突，并选择更稳妥的方案。

## 验证规则

根据技术栈选择：

- 后端变更：模型、服务、API 测试。
- 前端变更：构建和关键页面冒烟测试。
- 关键业务流程：集成测试或浏览器手工验证。
- 影响分发：打包验证。

## 每轮输出

每次迭代结束都要说明：

- 完成的需求条目。
- 修改的文件。
- 执行的测试或构建。
- 剩余 P0/P1/P2。
- 下一批建议。
