# 通用 GitHub 同步与 PR 合并

当用户要求“上传到 GitHub”“更新之前的仓库”“创建私密仓库”“用 WebBridge 打开 GitHub 上传”“更新 README 后推送”时，使用这个 skill。

## 核心原则

- 默认不直接强推 `main`。
- 远程有新提交时，先拉取并合并或 rebase，再推送。
- 推送失败时优先推新分支并创建 PR。
- Git / GitHub API 是主通道；WebBridge 只是应急通道。
- WebBridge 适合操作 GitHub 网页、确认登录、创建 PR、点击合并和解决少量冲突，不适合批量网页上传源码。
- 合并前要确认测试、构建、文档更新状态。

## 标准流程

1. 确认仓库状态：
   - `git status --short --branch`
   - `git remote -v`
   - `git log --oneline --decorate -5`
2. 检查是否有未提交改动：
   - 只提交本轮相关文件。
   - 不混入临时记录、日志、数据库、打包产物。
   - 如果用户明确说某类目录不要上传，例如本地 skills、测试数据库、release 产物，必须排除。
3. 更新文档：
   - README
   - 使用说明
   - 应用介绍
   - Release 或打包说明
4. 本地验证：
   - 根据项目类型运行测试。
   - 根据项目类型运行构建。
   - 如果用户要求桌面包或发版，确认打包产物。
5. 推送策略：
   - 先尝试 `git fetch origin main`。
   - 如果远程 `main` 有新提交，合并或 rebase 后再推。
   - 优先推送到工作分支，例如 `codex/<topic>`。
   - 创建 PR，而不是强推覆盖 `main`。
6. PR 合并：
   - 如果 GitHub 显示可自动合并，确认合并。
   - 如果显示冲突，优先本地解决并推回分支。
   - 如果 Git 推送不可用且冲突很少，可用 WebBridge 打开 GitHub 冲突编辑器解决。
7. 合并后同步本地：
   - `git fetch origin main`
   - 对齐本地 `main`。
   - 确认 `git status --short --branch` 干净。

## WebBridge 兜底规则

使用 WebBridge 操作 GitHub 时：

- 先确认桥接服务和浏览器扩展已连接。
- GitHub 网页上传文件 input 可能被浏览器安全策略拦截，不要把它作为批量源码上传主路径。
- 如果命令行 GitHub 网络不可用，但用户坚持用 WebBridge 同步，必须缩小范围，只处理本轮明确文件。
- 适合用 WebBridge 做：
  - 打开仓库或 PR。
  - 创建 PR。
  - 填 PR 描述。
  - 点击合并按钮。
  - 在冲突编辑器中处理少量文本冲突。
  - 极少量文件的网页编辑器应急提交。
- GitHub 网页表单输入中文可能在快照或标题里显示乱码；PR 标题和 merge 标题可使用 ASCII 安全标题，正文文件内容仍以仓库文件为准。

## WebBridge 应急提交流程

只有在 Git 推送、GitHub CLI 或 GitHub API 均不可用时使用：

1. 先列出本轮需要同步的文件清单。
2. 逐个打开 GitHub 文件编辑页。
3. 用网页编辑器替换文件内容。
4. 提交前确认页面识别到变更。
5. 使用简短 ASCII commit 标题，避免网页标题乱码。
6. 每个文件提交后，打开 raw 文件地址校验关键 marker。
7. 不要把本地未确认变更、临时文件或用户明确排除的目录带上去。
8. 完成后汇报哪些文件已校验，哪些文件未能同步。

如果需要同步的文件很多，应该停止并提示用户修复 Git 网络、代理或 token，而不是继续用网页编辑器逐个文件上传。

## 冲突处理规则

- 冲突前先读取两边内容，不要盲目选择一边。
- README 或说明文档冲突通常要整合两边内容。
- 代码冲突必须重新跑相关测试。
- 解决冲突后确认没有冲突标记：
  - `<<<<<<<`
  - `=======`
  - `>>>>>>>`

## 验证清单

- 本地测试通过。
- 本地构建通过。
- 需要打包时，release 产物存在。
- PR 已创建或已合并。
- 远程 `main` 已包含本轮提交。
- 本地分支已和远程主线对齐。
- 工作区没有临时文件和未提交变更。
 - 如果使用 WebBridge，应通过 raw 文件地址校验远端真实内容。

## 汇报格式

最终汇报包含：

- GitHub 仓库或 PR 链接。
- 已同步的提交或分支。
- 测试和构建结果。
- 是否已合并到 `main`。
- 剩余风险，例如网络不稳定、CI 未配置、Release 附件未上传。
