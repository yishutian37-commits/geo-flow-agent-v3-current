# GEO Flow Agent V2.3 详细使用说明

本文面向实际使用者，说明 GEO Flow Agent V2.3 的安装、初始化、资料录入、问题库生成、内容生成、发布记录、监测分析、报告中心和浏览器插件配置。

当前项目目录：

```text
D:\新版GEO优化项目\aio-marketing-suite\geo-flow-agent-v2\geo-flow-agent-v2
```

当前桌面应用包目录：

```text
D:\新版GEO优化项目\aio-marketing-suite\geo-flow-agent-v2\geo-flow-agent-v2\release
```

## 1. 这个 Agent 是做什么的

GEO Flow Agent 是一个本地运行的 GEO/AIO 生成式引擎优化工作台，核心目标是帮助企业完成：

1. 建立项目和品牌资料。
2. 把企业资料沉淀成可审核的品牌事实。
3. 诊断资料缺口。
4. 生成用户可能询问 AI 的问题库。
5. 生成内容任务和平台化文章草稿。
6. 记录发布渠道和发布结果。
7. 通过浏览器插件自动检测 AI 产品的真实回答。
8. 抓取原始回答、截图和信息来源。
9. 生成单次或多批次聚合报告。
10. 根据人工修改沉淀行业问题模板。
11. 根据检测短板继续补内容、复测和优化。

## 4. 配置大模型 API

文章生成、事实提取、问题库生成、反馈记忆分析都需要配置大模型。

操作路径：

```text
左侧菜单 -> AI 模型
```

建议步骤：

1. 新增或编辑模型。
2. 填写供应商、模型名称、API Key、Base URL。
3. 点击测试。
4. 测试通过后设为默认模型。

说明：

- 当前系统兼容 OpenAI 风格接口。
- 如果你的接口需要 `/v1` 后缀，系统会尽量自动兼容；一般 Base URL 可以填写根地址。
- 文章生成接口已放宽到较长超时，但大模型太慢时仍建议换更快模型或缩短文章要求。
- 视觉识别模式需要模型具备图片理解能力；不具备图片能力的模型只适合文本抓取模式。

## 5. 创建项目

操作路径：

```text
项目管理 -> 新建项目
```

建议填写：

- 项目名称：企业或品牌名称。
- 行业：企业所属行业。
- 地区：服务地区或目标市场。

## 6. 添加企业资料和品牌事实

品牌事实是整套系统的基础。文章生成、发布检查和报告引用都应尽量基于已确认事实。

操作路径：

```text
品牌事实库
```

推荐流程：

1. 选择项目。
2. 使用批量资料/AI 提取入口。
3. 一次性粘贴企业资料，例如：
   - 企业介绍
   - 资质证书
   - 证书编号
   - 地址
   - 联系方式
   - 产品/课程信息
   - 价格
   - 案例
   - 荣誉
   - 官网、公众号、媒体报道
4. 点击 AI 提取。
5. 系统会把资料拆成候选事实。
6. 人工检查每条事实。
7. 对准确、可公开使用的事实点击确认。

重要概念：

- `draft`：待确认事实，不能直接作为正式公开依据。
- `confirmed`：已确认事实。
- `public`：可公开使用。
- `restricted`：限制使用，不应进入公开文章。

如果文章生成提示“当前项目没有可公开使用的已确认品牌事实”，说明该项目还缺少 `confirmed + public` 事实，需要先确认基础事实。

## 8. 生成问题库

操作路径：

```text
项目详情 -> 问题库 -> 生成问题库
```

问题库会结合：

- 项目行业
- 行业问题模板
- 服务地区
- 品牌名称
- 已确认品牌事实
- 资质、价格、地址、产品、案例
- 用户画像和购买/验证场景

问题按 GEO 层级组织：

- 曝光/推荐层
- 验证/口碑层
- 转化/承接层
- 权威/对比层

每条问题包含：

- 问题文本
- 问题类型
- 标签
- 优先级
- 是否启用
- 是否重点关注
- 关键词拆解
- 问题公式
- 商业价值
- 证据支撑
- 内容可执行性
- 推荐发布平台

如果问题生成过于宽泛，优先补充品牌事实库，再重新生成问题库。

### 8.1 行业问题模板和自动沉淀

系统会从行业模板库读取当前行业的通用经验，例如：

- 这个行业的问题主体应该叫“供应商”“品牌”“门店”还是“服务商”。
- 这个行业常见的可信验证问法是什么。
- 哪些词不应该出现在该行业的问题里。
- 哪些问题是正向样例，后续可以学习。
- 哪些问题是反向样例，后续应该避免。

人工修改问题后，系统会自动记录调整事件：

- 新增问题：可能成为正向样例。
- 改写问题：记录改写前后，用于分析更好的问法。
- 禁用问题：可能成为反向样例。
- 删除问题：可能成为反向样例，并提取可复用禁用词。

这些记录不会直接自动改模板。管理员需要进入：

```text
系统设置 -> 问题模板优化建议
```

在这里查看系统汇总出的建议，例如新增禁用词、正向样例、反向样例。确认后才会写入行业模板库，后续同一行业的新项目或重新生成问题库时会复用这些经验。

建议使用方式：

1. 先正常生成问题库。
2. 删除明显不适合本行业的问题。
3. 把好问题改得更贴近真实客户提问。
4. 进入系统设置查看模板优化建议。
5. 确认写入。
6. 后续同类项目再生成问题库时，质量会逐步提高。

## 9. 生成内容任务和文章

操作路径：

```text
内容管理
```

推荐流程：

1. 先在项目详情中生成问题库。
2. 从问题矩阵生成内容任务。
3. 进入内容管理。
4. 选择内容任务。
5. 点击生成草稿。
6. 选择内容类型和发布平台。
7. 系统会结合品牌事实、问题库、平台规则和记忆库生成草稿。
8. 到“稿件草稿”中查看文章。

当前文章生成会参考：

- 已确认公开事实
- 任务关联问题
- 问题层级和用户意图
- 平台规则
- 记忆库写作规则
- 上一版草稿和反馈

注意：

- 草稿标题会单独保存到标题字段，正文不应再残留 JSON。
- 如果模型返回 JSON 或占位符，通常是模型不稳定或事实不足，建议补事实并重新生成。
- 如果生成失败提示 Network Error，优先检查模型 API、Base URL、超时和网络。

## 10.1 系统设置里的行业模板维护

```text
系统设置 -> 行业问题模板
```

这里可以直接编辑行业模板 JSON。适合高级维护人员使用，主要用于：

## 11. 记录发布结果

操作路径：

```text
内容管理 -> 稿件草稿 -> 记录发布
```

推荐流程：

1. 找到已经发布的文章草稿。
2. 点击记录发布。
3. 选择一个或多个发布平台。
4. 如果列表没有对应平台，可以手动输入平台名称。
5. 填写发布链接、发布时间、备注。
6. 保存发布记录。

说明：

- 一篇文章可以记录多个发布平台。
- 如果系统提示敏感词、行业风险或高风险状态，可以选择停止保存，也可以在确认真实发布后选择继续保存。
- 继续保存不代表内容完全合规，只是保留真实发布动作，方便后续复盘和复测。

## 12. 记忆库怎么用

操作路径：

```text
左侧菜单 -> 记忆库
```

记忆库用于保存项目写作偏好，例如：

- 文章要更口语化。
- 不要出现 AI 味。
- 标题不要夸张。
- 字数不要低于 1500 字。
- 多写本地场景。
- 多写资质、证据和案例。

使用方式：

1. 在草稿详情中提交修改反馈。
2. 系统先让 AI 分析反馈。
3. 反馈会被转成结构化规则。
4. 规则进入记忆库。
5. 后续文章生成和改写会参考这些规则。

## 13. 安装 QWebBridge 浏览器插件

监测分析需要浏览器插件。它的作用是让应用控制真实浏览器：

- 打开 AI 产品网页
- 输入问题
- 点击发送
- 等待回答
- 抓取回答文本
- 保存截图
- 抓取信息来源链接

### 13.1 插件目录

必须选择这个目录：

```text
D:\新版GEO优化项目\aio-marketing-suite\geo-flow-agent-v2\geo-flow-agent-v2\_external\QWebBridge\packages\extension\dist
```

不要选择：

- `_external\QWebBridge\packages\extension`
- `qwebbridge-extension-fixed`
- 其他临时目录

如果选错目录，Chrome/Edge 会提示：

```text
清单文件丢失或不可读取
```

### 13.2 Chrome 安装方法

1. 打开 Chrome。
2. 地址栏输入：

```text
chrome://extensions
```

3. 打开右上角“开发者模式”。
4. 点击“加载已解压的扩展程序”。
5. 选择 `extension\dist` 目录。
6. 确认扩展列表出现 QWebBridge。
7. 固定扩展图标。

### 13.3 Edge 安装方法

1. 打开 Edge。
2. 地址栏输入：

```text
edge://extensions
```

3. 打开“开发人员模式”。
4. 点击“加载解压缩的扩展”。
5. 选择 `extension\dist` 目录。
6. 确认扩展安装成功。

### 13.4 启动 QWebBridge 服务

项目目录下有启动脚本：

```text
启动QWebBridge-10087.cmd
```

使用方式：

1. 双击启动脚本。
2. 保持命令行窗口打开。
3. 打开健康检查：

```text
http://127.0.0.1:10087/health
```

如果健康检查失败，确认：

- 启动脚本是否已运行。
- 端口 10087 是否被占用。
- 浏览器扩展是否已加载。
- 当前 AI 网页是否已登录。

## 14. Kimi WebBridge

Kimi WebBridge 是另一个浏览器桥接能力，默认端口是：

```text
10086
```

健康检查：

```powershell
$exe=Join-Path $env:USERPROFILE '.kimi-webbridge\bin\kimi-webbridge.exe'
& $exe status
```

正常状态：

```json
{
  "running": true,
  "extension_connected": true
}
```

如果状态异常：

- `running:false`：启动 daemon。
- `extension_connected:false`：打开浏览器并确认扩展已安装。
- PID 存在但 HTTP 探测失败：可能是旧进程残留，需要清理 PID 或重启服务。

## 15. 监测分析怎么用

操作路径：

```text
左侧菜单 -> 监测分析
```

推荐流程：

1. 选择项目。
2. 选择目标 AI 产品，例如豆包、文心一言、Kimi。
3. 确认目标网页已登录。
4. 选择问题。
5. 选择识别模式：
   - 文本抓取：默认模式，速度更快。
   - 视觉识别：适合网页结构不稳定或文本抓取不完整的页面，需要多模态模型。
6. 设置等待时间。
7. 点击开始检测。
8. 检测过程中可以手动停止。
9. 检测完成后查看明细。

检测结果会统计：

- 品牌是否被提及
- 是否被推荐
- 情绪/风险
- 信息来源数量
- 原始回答
- 回答截图
- 来源链接

### 15.1 检测明细怎么看

检测明细按「问题 × 平台」展示。

每条明细包含：

- 问题
- 平台
- 是否提及品牌
- 是否推荐
- 情绪
- 信息来源数
- 补发建议
- 检测时间
- AI 搜索详情
- 删除
- 生成任务

如果某条问题没有提及品牌或没有被推荐，可以点击“生成任务”，系统会把它转为内容任务，后续补文章。

### 15.2 AI 搜索详情

详情页会展示：

- 问题
- 平台
- 原始回答
- 是否提及品牌
- 是否推荐品牌
- 判断依据
- 情绪判断
- 信息来源列表
- 回答截图
- 检测时间

信息来源不是系统编造的，而是从 AI 回答区域中的链接抓取并保存。

## 18. 常见问题

### 18.1 文章生成失败：Network Error

优先检查：

- 默认模型是否配置。
- API Key 是否有效。
- Base URL 是否正确。
- 模型响应是否超时。
- 本地网络是否可访问模型服务。

### 18.2 文章还是 JSON 格式

可能原因：

- 模型没有遵守输出格式。
- 事实库不足，模型倾向于返回结构化草稿。
- 平台或内容类型选择不合理。

建议：

- 补充并确认公开事实。
- 换更稳定的模型。
- 重新生成草稿。

### 18.3 视觉识别提示没有返回可解析 JSON

可能原因：

- 当前模型不支持图片理解。
- 截图区域没有完整回答。
- 模型输出不是 JSON。

建议：

- 使用具备视觉能力的模型。
- 增加等待时间。
- 优先使用文本抓取模式，只有文本抓取不稳定时再开视觉识别。

### 18.4 WebBridge 返回 502

优先检查：

- 本地桥接服务是否启动。
- 浏览器扩展是否连接。
- 目标 AI 网页是否登录。
- 页面是否还在验证或加载。
- selector 是否失效。

### 18.5 检测太快，抓不到完整回答

建议：

- 增加等待时间。
- 使用回答稳定检测。
- 检查目标 AI 页面是否仍在生成。
- 必要时使用视觉识别模式。

### 18.6 系统设置加载用户失败

旧版本 `.local` 邮箱会被 Pydantic 判定为保留域名。当前版本会自动修复为：

```text
pm01@geoflow.app
```
